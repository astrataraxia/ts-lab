# 프로젝트 아키텍처

이 문서는 특정 시점의 파일 목록을 고정하기 위한 문서가 아닙니다.
화면과 기능이 계속 추가되더라도 프로젝트가 지나치게 복잡해지지 않도록, 디렉터리 구조를 선택하는 기준과 지켜야 할 설계 계약을 정의합니다.

구현 기술이나 파일 이름은 바뀔 수 있지만, 기능을 분리하고 각 기능의 책임과 생명주기를 명확하게 관리한다는 방향은 유지합니다.

## 1. 아키텍처의 목표

이 프로젝트는 하나의 앱 안에 여러 인터랙티브 기능을 담는 것을 전제로 합니다.
현재는 소리 기반 시각화가 중심이지만, 앞으로 게임이나 다른 실험적인 화면이 추가될 수 있습니다.

따라서 구조의 목표는 다음과 같습니다.

- 새로운 화면을 추가할 때 기존 화면의 내부 코드를 수정하지 않도록 합니다.
- 오디오 입력, 데이터 처리, 화면 표현을 서로 교체할 수 있도록 합니다.
- 브라우저 API와 Canvas 같은 외부 기술이 화면 전체로 퍼지지 않도록 합니다.
- 한 기능의 변경이 다른 기능의 동작과 빌드에 미치는 영향을 줄입니다.
- 각 파일이 무엇을 소유하고 언제 정리해야 하는지 알 수 있도록 합니다.

## 2. 핵심 설계 철학

### 기능을 세로로 나눈다

폴더를 기술 종류만으로 나누기보다, 사용자가 접근하는 기능 단위로 먼저 나눕니다.

```text
pages/
├─ home/
├─ sound-canvas/
└─ game-lab/
```

`sound-canvas` 안에는 해당 기능에만 필요한 오디오 처리와 시각화 코드가 함께 있을 수 있습니다.
이렇게 하면 기능을 이해할 때 여러 상위 폴더를 오갈 필요가 줄어듭니다.

### 기능 내부에서는 책임을 가로로 나눈다

기능 하나 안에서는 역할을 다시 분리합니다.

```text
<feature>/
├─ page 또는 components/   화면 구조와 UI
├─ controller 또는 use-case/  이벤트와 흐름 조정
├─ domain 또는 logic/       기능의 핵심 규칙
├─ services 또는 adapters/  브라우저·외부 API 연결
├─ renderers/               화면 표현
└─ types.ts                 기능 내부 계약
```

모든 기능이 이 폴더를 전부 가져야 하는 것은 아닙니다.
작은 기능은 `page.ts`와 `controller.ts`만으로 시작하고, 코드가 실제로 커질 때만 하위 폴더를 추가합니다.

### 조립하는 곳과 일하는 곳을 분리한다

페이지를 시작하고 여러 모듈을 연결하는 코드는 조립 계층에 둡니다.
실제 오디오 계산, 게임 규칙, 렌더링 수식 같은 일은 각각의 책임 모듈이 수행합니다.

```text
화면 UI
  ↓ 이벤트
기능 controller / use-case
  ↓ 호출
기능 로직·서비스·렌더러
  ↓ 결과
화면 업데이트
```

controller가 모든 로직을 직접 가지고 있으면 처음에는 빠르지만, 기능이 늘어날수록 테스트와 교체가 어려워집니다.
controller는 “무엇을 언제 호출할지”를 관리하고, “어떻게 계산하는지”는 하위 모듈에 위임합니다.

## 3. 전체 디렉터리 구조

아래 구조는 현재와 미래의 기능을 함께 표현한 설계 템플릿입니다.
모든 파일과 폴더를 처음부터 만들 필요는 없으며, 기능이 생길 때 해당 기능의 경계 안에서 확장합니다.

```text
src/
├─ main.ts                         앱 시작점
├─ app/                            앱 전체 조립 계층
│  ├─ router.ts                    화면 선택과 전환
│  ├─ app-shell.ts                 전역 레이아웃이 필요할 때
│  └─ pages/                       사용자에게 보이는 기능 단위
│     ├─ home/
│     │  ├─ page.ts
│     │  └─ ...
│     ├─ sound-canvas/
│     │  ├─ page.ts
│     │  ├─ controller.ts
│     │  ├─ audio/
│     │  ├─ visualizer/
│     │  └─ ...
│     └─ game-lab/
│        ├─ page.ts
│        ├─ controller.ts
│        └─ ...
├─ shared/                         둘 이상의 기능이 실제로 공유하는 코드
│  ├─ types/
│  ├─ ui/
│  ├─ audio/
│  └─ utils/
└─ styles/
   ├─ global.css                   앱 전체의 토큰·기본 스타일
   └─ ...                          기능별 스타일이 필요할 때
```

### 각 영역의 의미

#### `main.ts`

애플리케이션의 진입점입니다.

- 전역 스타일과 앱 초기화를 시작합니다.
- 앱 루트를 찾습니다.
- 라우터 또는 앱 셸을 시작합니다.

`main.ts`에는 특정 화면의 오디오 처리, 게임 규칙, Canvas 렌더링을 넣지 않습니다.
이 규칙이 있어야 앱의 실행 방식이 바뀌어도 기능 코드를 그대로 재사용할 수 있습니다.

#### `app/`

앱 전체를 조립하는 계층입니다.
어떤 화면을 보여줄지, 공통 레이아웃을 어떻게 유지할지, 화면을 바꿀 때 무엇을 정리할지를 관리합니다.

`app/`은 개별 기능의 내부 구현을 직접 다루기보다, 각 기능이 제공하는 진입점만 사용해야 합니다.

#### `app/pages/`

사용자가 접근하는 기능을 보관합니다.
페이지는 단순한 URL 화면일 수도 있고, 하나의 큰 도메인 기능일 수도 있습니다.
중요한 기준은 “이 코드는 어느 기능에 속하는가?”입니다.

예를 들어 소리 입력과 Canvas 렌더링이 현재 Sound Canvas에서만 쓰인다면 다음처럼 기능 안에 둡니다.

```text
pages/sound-canvas/audio/
pages/sound-canvas/visualizer/
```

나중에 여러 기능에서 같은 오디오 분석 기능을 사용하게 되면 그때 `shared/audio/`로 승격할 수 있습니다.

#### `shared/`

공통이라는 이유만으로 미리 모아두는 저장소가 아닙니다.
둘 이상의 기능에서 실제로 사용되고, 어느 한 기능의 개념에 종속되지 않는 코드만 둡니다.

`shared`에 들어간 코드는 특정 페이지를 import하지 않아야 합니다.
반대로 페이지는 `shared`의 안정된 기능을 사용할 수 있습니다.

#### `styles/`

앱 전체에서 공유되는 색상 토큰, 폰트, 기본 여백, reset 등을 관리합니다.
특정 화면에만 필요한 스타일은 해당 feature 안에 둘 수 있습니다.
스타일 파일을 하나로 유지할지 기능별로 나눌지는 규모와 충돌 정도에 따라 바꿀 수 있습니다.

## 4. 기능 내부의 표준 역할

파일 이름은 달라질 수 있지만, 다음 역할은 가능하면 분리해서 유지합니다.

### 화면 표현 계층

`page.ts`, `components/`, `views/` 등이 여기에 해당합니다.

- DOM 구조를 생성합니다.
- 버튼, 입력, Canvas 같은 화면 요소를 노출합니다.
- 사용자가 보는 상태를 표시합니다.

화면 표현 계층은 오디오의 주파수 계산이나 게임 점수 계산을 직접 담당하지 않습니다.

### 흐름 조정 계층

`controller.ts`, `use-case/`, `store/` 등이 여기에 해당합니다.

- 사용자의 이벤트를 받습니다.
- 서비스를 호출합니다.
- 필요한 상태를 관리합니다.
- 화면을 시작하고 종료합니다.

이 계층은 기능을 연결하는 역할을 하며, 세부 알고리즘이 집중되는 곳이 되어서는 안 됩니다.

### 핵심 로직 계층

`domain/`, `logic/`, `analyzers/`, `rules/` 등이 여기에 해당합니다.

- 입력을 의미 있는 데이터로 변환합니다.
- 브라우저 DOM에 의존하지 않는 계산을 수행합니다.
- 가능한 경우 순수 함수로 작성합니다.

예를 들어 오디오의 저음·중음·고음 값을 계산하거나 게임의 판정 규칙을 계산하는 코드는 이 계층에 적합합니다.

### 어댑터·서비스 계층

`services/`, `adapters/`, `capture/` 등이 여기에 해당합니다.

- `AudioContext`, `MediaStream`, `getDisplayMedia()` 같은 브라우저 API를 감쌉니다.
- 파일, 네트워크, 외부 SDK와 연결합니다.
- 외부 기술의 형태를 내부에서 사용할 수 있는 형태로 변환합니다.

브라우저 API를 페이지나 렌더러 곳곳에서 직접 호출하지 않고 이 계층에 모으면, 입력 방식이나 실행 환경을 바꾸기 쉬워집니다.

### 렌더링 계층

`renderers/`, `canvas/`, `webgl/` 등이 여기에 해당합니다.

- 이미 준비된 데이터를 시각적인 결과로 바꿉니다.
- DOM 이벤트, 파일 선택, 오디오 권한 요청을 담당하지 않습니다.
- 렌더링 기술이 Canvas에서 WebGL이나 SVG로 바뀌어도 상위 계층의 데이터 계약은 유지하도록 합니다.

## 5. 의존 방향

의존성은 화면과 외부 기술이 핵심 로직을 오염시키지 않는 방향으로 흐르게 합니다.

```text
app 조립
  ↓
feature의 화면·controller
  ↓
feature의 로직·서비스·렌더러
  ↓
브라우저 API 또는 외부 시스템
```

일반적인 규칙은 다음과 같습니다.

- `main.ts`는 기능 내부 모듈을 직접 조작하지 않습니다.
- 페이지는 다른 페이지의 내부 파일을 직접 import하지 않습니다.
- 렌더러는 오디오 입력 방식이나 DOM 구조를 알지 않습니다.
- 핵심 로직은 DOM, Canvas, `window`에 직접 의존하지 않도록 합니다.
- 공통 모듈은 특정 페이지를 의존하지 않습니다.
- 기능 내부의 세부 구현은 기능 바깥에서 직접 import하지 않습니다.

페이지 사이에 데이터를 공유해야 한다면, router·app 상태·shared 계약 등 명확한 경로를 선택합니다.
한 페이지의 변수를 다른 페이지가 직접 가져오는 방식은 사용하지 않습니다.

## 6. 변경 가능한 부분

아래 항목은 프로젝트가 성장하면서 바뀔 수 있습니다.
변경 자체보다, 변경이 기능 경계를 침범하지 않는지가 중요합니다.

### 라우팅 방식

현재 hash 기반 라우팅을 사용하더라도, 이후 History API 기반 라우터나 라우팅 라이브러리로 바꿀 수 있습니다.
페이지가 URL의 구체적인 구현 방식에 의존하지 않게 하면 라우터 교체가 쉬워집니다.

### 입력 방식

Sound Canvas의 입력은 로컬 파일, 탭 오디오, 마이크, 네트워크 스트림 등으로 늘어날 수 있습니다.
입력 방식마다 어댑터를 추가하되, 분석 계층에는 동일한 의미의 오디오 데이터로 전달합니다.

### 렌더링 기술

Canvas 2D를 계속 사용할 수도 있고, WebGL·SVG·DOM 기반 표현을 추가할 수도 있습니다.
렌더러가 받는 데이터 계약을 유지하면 시각화 기술을 교체할 수 있습니다.

### 시각화 모드

새로운 모드는 기존 렌더러와 독립된 파일 또는 모듈로 추가합니다.
모드 목록을 관리하는 registry가 커지면 registry를 별도 파일로 분리할 수 있습니다.

### 상태 관리

기능이 작을 때는 controller 내부 상태로 충분합니다.
상태가 복잡해지면 feature 전용 store를 도입할 수 있고, 여러 화면이 같은 상태를 공유할 때만 app 또는 shared 수준으로 올립니다.

### 스타일 구성

초기에는 전역 스타일 하나로 시작할 수 있습니다.
화면별 스타일 충돌이 늘어나면 feature별 stylesheet, CSS module, 디자인 토큰 시스템 등으로 바꿀 수 있습니다.

## 7. 확장 시나리오

### 새 탭 또는 새 화면을 추가할 때

새 기능은 기존 기능의 하위 폴더가 아니라 새로운 feature로 시작합니다.

```text
pages/
├─ home/
├─ sound-canvas/
└─ game-lab/
```

일반적인 작업 순서는 다음과 같습니다.

1. 새 feature의 화면 진입점과 생명주기를 만듭니다.
2. 기능 내부에서만 사용하는 로직과 서비스를 feature 안에 둡니다.
3. router가 사용할 feature 진입점을 등록합니다.
4. home이나 공통 내비게이션에 링크를 연결합니다.
5. 화면을 떠날 때 이벤트, 타이머, 애니메이션, 스트림을 정리합니다.

새 화면을 만들기 위해 기존 Sound Canvas의 내부 파일을 복사하거나 직접 참조하지 않습니다.

### 오디오 입력을 추가할 때

새 입력 방식은 `capture` 또는 `adapters` 역할의 모듈로 추가합니다.
분석기와 렌더러의 계약을 유지하면 기존 시각화 모드는 수정하지 않고 입력만 늘릴 수 있습니다.

```text
입력 어댑터
  ↓
공통 오디오 분석
  ↓
표준 오디오 프레임
  ↓
시각화 렌더러
```

### 시각화 표현을 추가할 때

새 표현은 기존 표현과 분리된 renderer로 추가합니다.
렌더러는 `AudioFrame`, 화면 크기, 시간 등 필요한 입력만 받고, 파일 입력이나 권한 요청을 알지 않습니다.

### 공통 코드가 생겼을 때

처음에는 사용하는 feature 안에 둡니다.
두 번째 사용처가 실제로 생겼을 때 공통 개념을 추출하고 `shared`로 이동합니다.
이때는 한 feature의 내부 타입을 그대로 노출하기보다, 두 기능에 공통인 최소 계약을 새로 정의합니다.

## 8. 변경하면 안 되는 주요 계약

다음 항목은 구현 기술이 바뀌더라도 유지해야 하는 프로젝트의 핵심 규칙입니다.

### 앱 진입점은 얇게 유지한다

`main.ts`는 시작과 조립만 담당합니다.
기능 로직이 들어가기 시작하면 앱 전체와 기능의 결합도가 높아지므로, 로직은 반드시 feature 또는 shared로 이동합니다.

### 모든 화면은 명확한 생명주기를 가진다

화면은 시작할 때 만든 자원을 자신이 정리할 수 있어야 합니다.
권장되는 개념적 계약은 다음과 같습니다.

```ts
type PageInstance = {
  destroy: () => void;
};
```

실제 구현은 객체 대신 정리 함수를 반환할 수도 있지만, 의미는 동일해야 합니다.
화면 종료 시 다음 항목이 남지 않아야 합니다.

- `requestAnimationFrame`과 타이머
- DOM 이벤트 리스너
- 오디오 노드와 오디오 컨텍스트 연결
- `MediaStream` 트랙
- 임시 object URL

### 기능 경계를 넘어서 내부 구현하지 않는다

다른 페이지의 `controller`, `renderer`, 내부 상태를 직접 import하지 않습니다.
기능이 필요로 하는 공개 진입점이나 shared 계약을 통해서만 연결합니다.

### 데이터 계약은 입력 기술보다 우선한다

시각화 계층은 파일, 탭 캡처, 마이크 중 어떤 입력에서 데이터가 왔는지 알 필요가 없습니다.
입력 방식은 내부에서 바뀔 수 있지만, 분석 결과를 전달하는 표준 데이터 계약은 안정적으로 유지합니다.

현재 오디오 분석 결과처럼 데이터의 의미를 정의하는 타입은 다음 원칙을 따릅니다.

- 브라우저 객체나 DOM 요소를 직접 포함하지 않습니다.
- 값의 단위와 범위를 명확하게 합니다.
- 필드를 제거하거나 의미를 바꿀 때는 모든 소비자를 함께 검토합니다.
- 새 필드는 가능한 한 기존 소비자를 깨지 않는 방식으로 추가합니다.

### 렌더러는 데이터만 받아 표현한다

렌더러는 오디오 권한을 요청하거나 파일을 선택하지 않습니다.
렌더러의 책임은 입력받은 프레임을 화면에 표현하는 것까지입니다.

### 공유 영역은 의존성의 끝에 둔다

`shared`는 편의를 위한 전역 저장소가 아닙니다.
공유 코드는 특정 feature를 알지 않아야 하며, feature가 shared를 사용하는 방향을 유지합니다.

### 예외적인 결합은 문서화한다

성능이나 브라우저 제약으로 일반 규칙을 어겨야 할 때는 암묵적으로 처리하지 않습니다.
왜 결합이 필요한지, 어떤 범위에 영향을 주는지, 나중에 분리할 조건을 코드나 문서에 남깁니다.

## 9. 구조가 커질 때의 판단 기준

다음 질문을 순서대로 확인하면 폴더를 어디에 둘지 결정할 수 있습니다.

1. 이 코드는 사용자에게 보이는 하나의 기능에만 필요한가?
   - 그렇다면 해당 feature 안에 둡니다.
2. 이 코드는 화면과 무관한 핵심 계산인가?
   - 그렇다면 `logic`, `domain`, `services` 중 책임에 맞는 곳에 둡니다.
3. 둘 이상의 feature가 실제로 사용하는가?
   - 그렇다면 공통 부분만 `shared`로 추출합니다.
4. 브라우저나 외부 시스템에 직접 의존하는가?
   - 그렇다면 adapter 또는 service로 감쌉니다.
5. 화면이 종료될 때 정리해야 할 자원을 만드는가?
   - 그렇다면 생성한 모듈이 정리 책임도 가져야 합니다.

폴더의 이름보다 중요한 것은 한 방향의 책임과 의존 관계가 유지되는지입니다.

## 10. 검증 기준

구조나 기능을 변경한 뒤에는 최소한 다음을 확인합니다.

```powershell
npm run format
npm run check
npm run build
```

추가로 화면을 만들거나 브라우저 자원을 다루는 기능이라면 다음도 확인합니다.

- 화면을 여러 번 이동해도 이벤트와 애니메이션이 중복되지 않는가?
- 오디오 입력을 중지한 뒤 스트림과 오디오 노드가 정리되는가?
- 새 feature가 다른 feature의 내부 구현을 import하지 않는가?
- 공통으로 추출한 코드가 특정 페이지에 종속되지 않는가?
- 기존 입력 방식과 시각화 모드가 그대로 동작하는가?

이 문서는 구현이 바뀔 때마다 파일 목록을 기계적으로 갱신하기보다, 구조 원칙과 핵심 계약이 여전히 유효한지 검토하는 데 사용합니다.
